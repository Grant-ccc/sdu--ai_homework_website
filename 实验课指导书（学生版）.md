# 实验课指导书（学生版）

## 实验主题

使用 `OpenCode` 与免费开源模型完成一个小型软件项目的生成、验证、迭代与 AI 辅助代码审查

---

## 一、实验目标

完成本实验后，你应当能够：

1. 使用 `OpenCode` 配合免费开源模型生成一个可运行的小型代码项目。
2. 编写较清晰的需求 Prompt，并根据结果进行迭代调整。
3. 对 AI 生成代码进行运行验证、问题定位和基础修复。
4. 使用 AI 对项目进行结构化代码审查。
5. 对 AI 审查意见进行筛选、判断、补充和二次修改。
6. 完成一轮“AI 生成 + AI 审查 + 人工把关”的完整开发闭环。

---

## 二、实验背景

在前面的课程中，我们已经学习了：

1. 代码审查为什么重要。
2. 代码审查通常关注哪些维度。
3. AI 如何帮助我们更快地生成代码与辅助审查。
4. AI 的能力边界与人的最终责任。

这节实验课的重点不是“让 AI 帮你把代码全写完”，而是让你真正体验现代软件开发中的完整流程：

1. 明确需求。
2. 与 AI 协作生成项目。
3. 运行并验证结果。
4. 基于问题继续迭代。
5. 发起 AI 辅助代码审查。
6. 由人做最终判断与修改。

请记住一个核心原则：

> AI 可以帮助你写代码、看代码、找问题，但你仍然要对你最终提交的代码负责。

## 四、实验环境要求

请在实验开始前确认以下环境已经准备好：

1. 已安装并可以使用 `OpenCode`
2. 已配置至少一个免费可用的开源模型
3. 已安装 `Git`
4. 已安装 `Node.js` 或 `Python`
5. 已具备本地运行前端项目的能力
6. 已安装浏览器（Chrome、Edge、Safari 均可）

建议目录结构如下：

```text

├── project/
├── notes/
├── review/
└── report/
```

你也可以按自己的习惯组织目录，但建议至少区分：

1. 项目代码目录
2. Prompt 与交互记录目录
3. 审查记录目录
4. 实验报告目录

## 五、实验任务说明

本次实验要求你围绕一个统一主题，使用 `OpenCode` 与免费开源模型完成一个最小可用项目，并进行一轮 AI 辅助代码审查。

### 统一实验题目

请完成一个：

**课程作业提交管理小工具**

### 基础功能要求

项目至少应包含以下功能：

1. 添加课程作业
2. 为每条作业记录填写：
   - 课程名称
   - 作业标题
   - 截止日期
   - 完成状态
3. 支持删除作业
4. 支持筛选：
   - 按课程筛选
   - 按“已完成 / 未完成”筛选
5. 页面刷新后数据仍然保留

### 拓展要求

在基础功能上，每人需要自行加入 `1-2 个小扩展功能`。

可选示例：

1. 按截止日期排序
2. 作业优先级
3. 临近截止提醒
4. 搜索功能
5. 逾期任务高亮显示

建议不要添加过于复杂的后端功能。此次实验重点是：

1. 需求表达
2. AI 协作开发
3. AI 辅助代码审查
4. 人工判断与修改

## 七、详细实验步骤

### 阶段 1：环境检查与实验准备

#### 目标

确认你已经具备使用 `OpenCode` 和免费模型完成实验的能力。

4. 在实验目录中初始化项目工作区

#### 建议操作

可以先用一个非常简单的问题测试你的模型是否正常：

```text
请不要创建文件，只告诉我：如果我要实现一个最简单的 HTML 页面，你会如何组织文件结构？
```

如果模型可以正常返回结果，则说明环境基本可用。

#### 记录要求

请在 `notes/` 目录中记录：

1. 本人姓名
2. 使用的模型名称
3. 是否顺利完成环境测试
4. 如果遇到问题，问题是什么，如何解决

### 阶段 2：编写需求 Prompt

#### 目标

学习如何把“模糊想法”变成“可供 AI 执行的开发需求”。

#### 说明

很多同学第一次使用 AI 开发时，会直接说：

```text
帮我做一个作业管理系统。
```

这种 Prompt 太模糊，AI 很可能会：

1. 选错技术栈
2. 做得太复杂
3. 功能理解不完整
4. 生成你不熟悉、也不方便调试的代码

所以，你应该给 AI 一个**结构化需求描述**。

#### 推荐 Prompt 模板

请先使用以下模板，并根据你们组的扩展功能自行修改：

```text
你要帮我生成一个适合大学生使用的小型 Web 应用。

项目名称：
课程作业提交管理小工具

目标用户：
大学生

核心功能：
1. 添加课程作业
2. 设置课程名、作业标题、截止日期、完成状态
3. 支持按课程筛选
4. 支持按“未完成/已完成”筛选
5. 支持删除作业
6. 页面刷新后数据仍然保留

扩展功能：
1. [请在这里写入你们组自己的扩展功能]
2. [可选第二个扩展功能]

技术要求：
1. 使用前端单页应用实现
2. 优先使用 HTML/CSS/JavaScript 或 React
3. 不要依赖复杂后端
4. 界面尽量简洁清晰
5. 代码需要有合理的文件结构
6. 生成后可以直接本地运行

输出要求：
1. 先给出开发计划
2. 再创建项目文件
3. 最后说明如何启动和测试
```

#### 任务

1. 基于模板写出你们自己的 Prompt
2. 扩充 `1-2 个` 额外需求
3. 在调用 AI 之前，先人工检查一次 Prompt 是否清晰

#### 自查问题

在发给 OpenCode 之前，请先检查：

1. 目标是否明确？
2. 功能是否完整？
3. 技术栈是否受控？
4. 是否限制了项目复杂度？
5. 是否要求 AI 先给计划再写代码？

#### 记录要求

请保存：

1. 最终使用的 Prompt
2. 为什么这样写
3. 增加了哪些额外需求

---

### 阶段 3：使用 OpenCode 生成初始项目

#### 目标

利用 `OpenCode` 根据需求生成一个最小可用项目。

#### 原则

请不要一上来就让 AI 直接改完整个目录。更推荐的流程是：

1. 先让 AI 输出实现计划
2. 人工检查计划是否合理
3. 再让 AI 开始生成文件和代码

#### 推荐步骤

第一步：要求 AI 输出计划

```text
请先不要直接写代码。
请先根据我的需求，输出一个简洁的开发计划，包括：
1. 技术栈选择
2. 项目文件结构
3. 每个模块的职责
4. 你打算如何实现数据持久化
```

第二步：人工检查计划

重点看：

1. 是否引入了你们不会用的技术
2. 是否把原本简单的项目复杂化了
3. 是否遗漏了需求
4. 是否选择了合理的数据保存方式（如 localStorage）

第三步：让 AI 生成项目

示例：

```text
计划看起来可以。请开始创建项目。
要求：
1. 先实现最小可用版本
2. 保持结构简单
3. 不要加入与需求无关的依赖
4. 生成完成后告诉我如何启动项目
```

#### 运行项目

项目生成后，请尽快运行，验证是否能启动。

你需要检查：

1. 项目是否能成功安装依赖
2. 项目是否能正常启动
3. 页面是否可以打开
4. 是否已经具备至少基础功能框架

#### 如果运行失败怎么办

不要立刻推倒重来。优先使用如下方式：

```text
项目运行失败了。请基于以下报错信息做最小修改，不要重写整个项目。

报错信息：
[粘贴错误日志]

要求：
1. 解释错误原因
2. 只修改必要内容
3. 修改后告诉我改了哪些文件
```

#### 记录要求

记录以下内容：

1. AI 给出的计划摘要
2. 生成后的项目结构
3. 首次运行是否成功
4. 如果失败，报错是什么
5. 你们如何处理失败

---

### 阶段 4：运行、验证与迭代修改

#### 目标

验证 AI 生成项目是否真正满足需求，并完成一轮面向问题的迭代。

#### 核心认识

第一次生成的结果通常**不会完美**。

你需要像真正的软件工程师一样，去验证、定位问题、描述问题、推动修复，而不是只看“有没有生成出来”。

#### 功能检查清单

请逐项验证：

1. 是否可以新增作业
2. 新增后是否正确显示
3. 是否可以切换完成状态
4. 是否可以删除作业
5. 是否可以按课程筛选
6. 是否可以按完成状态筛选
7. 页面刷新后数据是否仍然保留
8. 新增的扩展功能是否工作正常

#### 界面检查清单

1. 页面结构是否清晰
2. 表单输入是否直观
3. 按钮文案是否明确
4. 日期信息是否易读
5. 是否存在明显布局错乱

#### 代码结构检查清单

1. 文件命名是否合理
2. 是否存在过长函数
3. 是否有明显重复代码
4. 是否存在难以理解的命名

#### 推荐迭代 Prompt

```text
请不要重写整个项目，只做最小修改来解决以下问题：
1. [问题 1]
2. [问题 2]
3. [问题 3]

要求：
1. 保持当前技术栈不变
2. 优先做最小修改
3. 修改后说明你改了哪些文件以及为什么
4. 如果某个问题与你理解的不一致，请先说明再修改
```

#### 示例问题

你可以参考如下表达方式：

1. 删除按钮点击后列表没有立即刷新
2. 截止日期显示格式不清楚
3. localStorage 中的数据结构过于混乱
4. 某个函数太长，难以维护
5. 未完成和已完成筛选逻辑有误

#### 记录要求

请记录：

1. 发现了哪些问题
2. 你如何向 AI 描述问题
3. AI 是否准确理解了问题
4. 修改后问题是否解决

---

### 阶段 5：发起 AI 辅助代码审查

#### 目标

将课堂上学到的 Code Review 知识应用到真实项目中。

#### 核心原则

这一阶段不是“让 AI 再看一遍代码就结束”，而是：

1. 让 AI 先做初审
2. 由人判断 AI 审查意见的质量
3. 再由人决定哪些要改、哪些不改、哪些 AI 漏掉了

#### 推荐审查 Prompt

请在项目基本可运行后，向 OpenCode 提交以下类似请求：

```text
请你作为代码审查者，对当前项目做一次结构化 Code Review。

请从以下五个维度给出意见：
1. 逻辑与正确性
2. 可读性与可维护性
3. 性能
4. 安全性
5. 最佳实践

要求：
1. 不要泛泛而谈
2. 尽量指出具体文件、函数或问题位置
3. 区分“必须修改”和“建议优化”
4. 给出修改建议，但暂时不要直接改代码
```

#### 输出要求

请让 AI 的输出尽量具备以下结构：

```text
[Must] 问题 1
- 所在位置：
- 问题描述：
- 影响：
- 修改建议：

[Suggestion] 问题 2
- 所在位置：
- 问题描述：
- 影响：
- 修改建议：
```

#### 你需要做的事情

请不要盲目接受 AI 的审查意见。你需要完成以下任务：

1. 从 AI 给出的审查意见中，选出至少：
   - `2 个必须修改的问题`
   - `1 个可讨论但不一定修改的问题`
2. 必须**人工补充至少 1 个 AI 没发现的问题**

#### 人工复审建议角度

如果你一时不知道该看什么，可以从下面几个角度继续检查：

1. 状态更新是否有边界问题
2. 本地存储是否有异常处理
3. 表单输入是否合法
4. 命名是否能表达真实含义
5. 是否存在明显重复逻辑
6. 是否存在难以维护的超长函数

#### 建议整理为表格

你们可以在 `review/` 目录中整理如下表格：

| 问题来源 | 问题描述 | 严重程度 | 是否修改 | 理由 |
|---|---|---|---|---|
| AI | 删除后页面未立即刷新 | 高 | 是 | 影响功能正确性 |
| AI | 建议统一变量命名 | 低 | 否 | 当前可读性可接受 |
| 人工 | localStorage 异常未处理 | 中 | 是 | 可能导致运行报错 |

#### 记录要求

请保留：

1. AI 原始审查输出
2. 你们的人工筛选结果
3. 你们人工补充的问题

---

### 阶段 6：根据审查结果修改并完成提交

#### 目标

完成“生成 -> 运行 -> 审查 -> 修改 -> 总结”的完整闭环。

#### 修改要求

请根据上一阶段的审查结果，至少完成：

1. 2 个必须修改的问题修复
2. 1 个人工补充问题修复

#### 推荐修改 Prompt

```text
请根据以下代码审查结果进行修改：
1. [必须修改问题 1]
2. [必须修改问题 2]
3. [人工补充问题]

要求：
1. 只修改必要内容
2. 不要重构无关模块
3. 修改后总结每项修改对应了哪条审查意见
```

#### 完成后请再次验证

修复后请重新检查：

1. 项目是否仍然可以运行
2. 原有功能是否仍然正常
3. 被指出的问题是否确实解决
4. 有没有引入新的问题

## 九、常见问题与建议

### 1. AI 生成的项目跑不起来怎么办？

建议顺序：

1. 先阅读报错
2. 把报错信息给 AI
3. 要求它做最小修复
4. 不要立刻重建整个项目

### 2. AI 总是修改太多怎么办？

你需要明确约束：

```text
请只做最小修改，不要重构无关模块。
```

### 3. AI 审查意见太空泛怎么办？

你需要提高要求：

```text
请指出具体文件、函数和问题位置，不要只给泛泛建议。
```

### 4. 如果 AI 没发现什么问题怎么办？

这本身就是实验的一部分。

你需要人工继续检查，并补充至少 1 个问题。

### 5. 可以完全相信 AI 吗？

不可以。

本实验最重要的目标之一，就是让你意识到：

> AI 可以帮助你提高效率，但最终的判断与责任仍然属于人。

---

## 十、实验中的行为建议

请尽量做到：

1. 不要只复制粘贴 Prompt，要理解自己为什么这么问
2. 不要只看 AI 是否“给出答案”，要看答案是否正确、是否可维护
3. 不要为了完成实验而盲目接受所有 AI 建议
4. 不要把代码审查理解为“再让 AI 看一遍”
5. 要主动比较：
   - AI 看到了什么
   - AI 没看到什么
   - 人为什么仍然重要

## 十一、课后思考题

请在实验结束后继续思考以下问题：

1. 如果 AI 可以写代码、审代码、修代码，那么程序员未来最核心的能力是什么？
2. 哪些代码你愿意交给 AI 快速生成，哪些代码你一定要自己严格把关？为什么？
3. 在你今天的实验中，AI 最像一个什么角色：实习生、同事、工具、还是自动化流水线？为什么？
4. 如果未来是“多个 Agent 并行开发”，人类工程师最重要的职责会变成什么？

## 十二、最后提醒

这节实验课的重点不是“谁做出的页面最好看”，也不是“谁的 Prompt 最花哨”。

重点是你是否真正理解了现代软件开发中的这个闭环：

1. 用 AI 加速开发
2. 用运行结果验证代码
